# 收集策略

> Agent 执行候选收集时的操作指引。不写死代码，Agent 用已有的 web_search / browser 工具执行。

## 渠道与搜索策略

候选收集采用多路召回。不要只搜一个关键词或一个平台。

- `Task Recall`：围绕用户当前任务生成 query。
- `Scenario Recall`：围绕长期场景/领域生成 query。
- `Gap Recall`：围绕当前 Agent 缺失能力生成 query。
- `Trend Recall`：围绕 trending、latest、popular、recently updated 生成 query。

每一路最多取 5 个候选，总候选控制在 15-25 个。合并后去重、过滤、排序，只推荐最高分 1 个。

### 1. ClawHub（优先级 5，最可信）

```
搜索词："site:clawhub.ai skills"
或浏览器打开 https://clawhub.ai → 找到 trending / latest skills 列表
```

提取要点：
- 每个 skill 走详情页获取 name, description, categories, updated_at
- 优先取 trending 前 5

### 2. MCP Market（优先级 4）

```
搜索词："site:mcpmarket.com trending skills"
或浏览器打开 https://mcpmarket.com → 找 trending / popular
```

提取要点：
- 获取 name, description, tags（映射为 categories）
- 抓取 install count（映射为 downloads）

### 3. Smithery（优先级 3）

```
搜索词："site:smithery.ai new skills"
或浏览器打开 https://smithery.ai → 找 recently added
```

提取要点：
- 获取 name, description, categories
- 抓取 stars, usage count

### 4. Glama（优先级 2）

```
搜索词："site:glama.ai mcp servers"
或浏览器打开 https://glama.ai/mcp → 找 popular / new
```

提取要点：
- 获取 name, description, tags
- 抓取 stars, downloads

### 5. GitHub（优先级 2）

```
搜索词："mcp server skill trending" OR "agent skill automation"
或搜索 site:github.com MCP server automation / AI agent skill
```

提取要点：
- 获取 repo name、description、stars、updated_at
- 优先选择有 README、安装说明和明确 license 的项目
- 标记 source=github
- GitHub 来源质量参差不齐，缺少安装文档或安全说明时至少 medium 风险

### 6. X/Twitter（manual only，至少 medium 风险）

X/Twitter 不作为 daily 默认来源。manual 探索时可以用原生搜索/浏览能力读取公开结果，但不要登录、不要绕过访问限制。

提取要点：
- 只作为补充线索，必须能追到项目主页或仓库
- 标记 source=x
- 没有 url 的纯讨论内容不进入候选

## 归一化规则

每个来源的原始数据归一化为标准 candidate：

```json
{
  "skill_id": "{source}:{name_lower}",
  "name": "原始名称",
  "source": "clawhub|mcpmarket|smithery|glama|github|x",
  "url": "skill 页面 URL",
  "source_code_url": "GitHub URL（如果有）",
  "description": "修剪到一句话",
  "categories": ["字符串数组"],
  "updated_at": "ISO 8601",
  "popularity": {
    "downloads": 0,
    "stars": 0
  },
  "cross_source_count": 在几个来源出现过,
  "trend": {
    "source_count": 1,
    "recent_mentions": 0,
    "updated_recently": true,
    "trend_reason": "近期在多个来源出现或增长较快"
  },
  "has_install_docs": true/false,
  "has_security_disclosure": true/false,
  "required_toolsets": ["从描述推断"],
  "complexity": "lightweight|framework|platform",
  "risk_level": "low|medium|high"
}
```

说明：
- `cross_source_count` 是可选辅助字段，脚本不会依赖它。
- `trend` 是可选辅助对象，用于记录网络热门证据；趋势只能加分，不能覆盖任务匹配和安全。
- 不要额外输出“元数据是否完整”这类汇总布尔字段；安全判断直接看 `source_code_url`、`has_install_docs`、`has_security_disclosure`。
- 不要额外输出社交点赞数字段；社交热度不进入标准 schema。

## 执行建议

- 每渠道取前 3-5 个，共 15-25 个候选
- 同一 skill 出现在多来源 → 按优先级取最高的，`cross_source_count` += 1
- 趋势候选必须同时满足生产力跃迁和安全要求；不要因为热门就推荐
- 耗时控制在 2 分钟内
- 如果某渠道不可访问（403/超时），跳过，不阻塞整体流程

## 降级策略

- 所有渠道不可用：返回空候选列表，本次沉默，不发推荐。
- 部分渠道可用：用可用渠道继续，降低候选数量但不停。
- 搜索结果不可解析：用该渠道的备用搜索词重试 1 次。
- 候选少于 3 个：降低置信度继续推荐，或不推荐；不要用固定候选池伪装实时发现。
- 当前 Agent 没有无审批搜索/浏览能力：manual 引导配置搜索/浏览能力；daily 走失败提醒。
